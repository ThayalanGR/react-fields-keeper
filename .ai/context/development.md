# Development Guide

## Quick Start

```bash
yarn install
npm run dev          # Dev server at localhost:5173
```

Navigate to `http://localhost:5173/#example-1` through `#example-39` to explore features interactively.

---

## Key Scripts

| Script | Command | What It Does |
|---|---|---|
| Dev server | `npm run dev` | Vite dev server (`--host`) for App.tsx demo |
| Demo build | `npm run build` | `tsc && vite build` — builds the demo app |
| Library build | `npm run build:library` | `vite build -c library.vite.config.ts` — builds `dist/` for npm |
| Lint | `npm run lint` | ESLint with zero warnings enforced (will fail CI on any warning) |
| Preview | `npm run preview` | Preview the demo build |
| Publish | `npm run publish:version` | `npm publish` — publishes to npm |

---

## Library Build Details

Config: `library.vite.config.ts`

- Entry: `src/index.ts`
- Outputs: `dist/index.es.js` (ESM) + `dist/index.umd.js` (UMD)
- Types: generated by `vite-plugin-dts` → `dist/index.d.ts`
- CSS: compiled from Less and **injected into JS** by `vite-plugin-css-injected-by-js` — no separate CSS import needed by consumers
- React and react-dom are **externalized** (peer deps, not bundled)

After `build:library`, the `dist/` folder is what gets published to npm.

## Adding a New Feature

1. **Read `.ai/context/`** first for architecture context.
2. Core logic lives in `src/FieldsKeeper/` — types go in `FieldsKeeper.types.ts`, component changes in the relevant `.tsx` file.
3. Add a new example file in `src/Examples/Example{N}_FeatureName.tsx` to demonstrate the feature.
4. Register the example in `src/Examples/index.tsx` and in `src/App.tsx`.
5. Export any new public types/components through `src/index.ts`.
6. Run `npm run lint` — zero warnings required before committing.

## Code Conventions

### TypeScript
- Strict mode enabled (`noUnusedLocals`, `noUnusedParameters`, `strictNullChecks`)
- All public interfaces prefixed with `I` (e.g. `IFieldsKeeperItem`)
- Type-only exports use `export type`
- Generic types default to `any` (e.g. `IFieldsKeeperItem<T = any>`)

### Component Patterns
- Components read state via `useFieldsKeeperStore(instanceId)` Zustand hook
- `instanceId` is read from React Context (`useContext(FieldsKeeperContext)`) — never passed as prop between internal components
- Renderer props follow the `*Renderer` naming convention and receive typed props objects
- Event handler props follow the `on*` convention

### CSS / Styling
- Less files, BEM-like naming with prefix `react-fields-keeper-`
- MS Fluent icon class names use prefix `fk-ms-Icon-`
- Theme colors are applied as inline CSS variables or direct style props — not hardcoded in Less

## CI / CD

`.github/workflows/npm-publish.yml` runs on version tags:
1. Installs dependencies (yarn)
2. Runs `build:library`
3. Publishes to npm with `NODE_AUTH_TOKEN`

There are **no automated tests** currently — quality is maintained via TypeScript strict mode, ESLint zero-warnings, and the 39 example files.

## State Management Patterns

When modifying state-related code in `src/FieldsKeeper/FieldsKeeper.context.ts`:

- The Zustand store is a **global object** keyed by `instanceId`
- Initialize/teardown happens in `FieldsKeeperProvider` via `useEffect`
- Never access store state directly outside of components — always use the `useFieldsKeeperStore` hook
- The `onUpdate` callback is the consumer's only reliable signal for state changes — keep it accurate

---

## Common Gotchas

- **instanceId collisions**: If two providers accidentally share an `instanceId`, their state will merge. Always pass unique IDs for nested providers.
- **CSS injection**: Since CSS is injected via JS, SSR setups need special handling. This is a known limitation.
- **acceptTypes** is checked on *drop* not on render — items don't visually grey out based on type compatibility before dragging.
- **maxItems** is enforced in `assignFieldItems` — exceeding it silently ignores the assignment.
- **`_fieldItemIndex`** is an internal field managed by the library. Setting it manually in `allItems` will be overwritten.
- Lint is strict — unused variables or imports will fail the CI pipeline.
